# description 최적화 가이드

출처: https://agentskills.io/skill-creation/optimizing-descriptions

## 트리거 작동 원리

시작 시 에이전트는 모든 스킬의 `name` + `description`만 로드한다 (각 ~100 토큰).
사용자 요청이 description과 일치하면 전체 SKILL.md를 컨텍스트에 로드한다.

description이 트리거의 전부다. description이 나쁘면 스킬은 존재해도 쓸모없다.

한 가지 주의사항: 에이전트는 스스로 처리할 수 있는 간단한 작업에는 스킬을 참조하지 않는다.
"이 PDF 읽어줘" 같은 단순 요청은 description이 완벽해도 트리거 안 될 수 있다.
특수 지식이 필요한 작업 — 생소한 API, 도메인 특화 워크플로우, 비일반적 포맷 —
에서 좋은 description이 빛을 발한다.

## 효과적인 description 원칙

### 1. 명령형 어조 사용
에이전트에게 언제 행동할지 지시하는 형태:

```yaml
# 좋음 — 언제 사용할지 명시
description: 사용자가 CSV 파일을 분석하고 싶을 때 이 스킬을 사용한다...

# 약함 — 기능 설명에 그침
description: 이 스킬은 CSV 파일을 분석한다...
```

### 2. 구현이 아닌 사용자 의도 중심

```yaml
# 나쁨 — 내부 구현 설명
description: pdfplumber와 pytesseract를 실행해 PDF 텍스트를 추출한다.

# 좋음 — 사용자가 하려는 일
description: PDF 텍스트 추출, 양식 작성, 파일 병합. PDF로 작업할 때 사용.
```

### 3. 키워드를 쓰지 않아도 트리거되도록

```yaml
description: >
  CSV, TSV, Excel 파일을 분석한다. 사용자가 데이터 파일로 탐색, 변환,
  시각화 작업을 하고 싶을 때 사용 — "CSV"나 "분석"을 명시하지 않더라도.
```

### 4. 인접 스킬과의 경계 명시 (오탐 방지)

비슷한 스킬이 있거나 혼동 가능성이 있으면 경계를 명확히:

```yaml
description: >
  DB 마이그레이션 파일 생성. 스키마 변경 시 사용.
  마이그레이션 실행, 롤백, 일반 SQL 쿼리에는 사용하지 않는다.
```

## 개선 전후 예시

```yaml
# 전 — 너무 좁음
description: CSV 파일 처리.

# 후 — 구체적이고 의도 중심, 키워드 무관
description: >
  CSV, TSV, Excel 데이터 파일을 분석한다 — 요약 통계 계산, 파생 컬럼 추가,
  차트 생성, 지저분한 데이터 정제. 사용자가 데이터 파일로 탐색, 변환,
  시각화를 원할 때 사용 — "CSV"나 "분석"을 명시하지 않더라도.
```

## Description 테스트 방법

약 10개의 테스트 쿼리를 설계한다:
- **should-trigger 5개**: 표현 방식, 명시성, 세부 수준을 다양하게
- **should-not-trigger 5개**: 키워드는 비슷하지만 실제론 다른 작업 (near-miss)

각 쿼리를 머릿속으로 실행: 이 description이 에이전트를 스킬로 유도할까?

**강한 near-miss 예시 (should-not-trigger):**
- CSV 분석 스킬: "CSV 읽어서 각 행을 postgres에 업로드하는 파이썬 스크립트 작성"
  (CSV 포함하지만 목적은 ETL, 분석 아님)
- 마이그레이션 스킬: "Excel 예산 스프레드시트의 수식 수정"
  (스프레드시트 개념 공유하지만 다른 도메인)

## 흔한 실패 패턴

| 문제 | 증상 | 해결 |
|------|------|------|
| 너무 좁음 | 다르게 표현한 요청에 트리거 안 됨 | "~라고 명시하지 않더라도" 추가 |
| 너무 넓음 | 관계없는 요청에도 트리거 됨 | 이 스킬이 하지 않는 것 명시 |
| 구현 중심 | 의도 기반 쿼리를 놓침 | 사용자가 이루려는 목표 중심으로 재작성 |
| 인접 케이스 누락 | 비슷하게 표현된 요청 실패 | 더 많은 트리거 시나리오 추가 |
