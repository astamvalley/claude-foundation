# auto

자연어 입력 → 가장 적합한 crb 커맨드 자동 탐지 → 실행.

```
/crb:auto "결제 로직에 SQL 인젝션 있을 것 같아"  →  security 실행
/crb:auto "파일 업로드 기능 구조 잡고 싶어"       →  mold 실행
/crb:auto "이 에러 왜 나는지 모르겠어"            →  debug 실행
/crb:auto "로그인 플로우 만들어줘"                →  forge 실행
```

## 입력 파싱

인수 없이 실행 시:
```
어떤 작업을 도와드릴까요? 자연어로 설명해 주세요.

예시:
  "결제 API 만들어줘"
  "이 코드 보안 문제 없는지 봐줘"
  "이 에러가 왜 나는지 모르겠어"
  "로그인 기능 설계해줘"
```

## 라우팅 우선순위

입력을 아래 우선순위로 분석한다:

```
Priority 1 — 특화 (가장 구체적인 키워드 먼저):
  보안/OWASP/취약점/인젝션/XSS      → security
  TDD/테스트 먼저/test-driven        → forge --tdd
  에러/버그/스택트레이스/왜 안됨     → debug
  리뷰/검토/코드 리뷰/품질           → assay

Priority 2 — 코어:
  설계/아키텍처/구조/스키마          → mold
  기획/방향/전략/아이디어/어떻게 할까 → cast
  반론/비판/검증/문제점/약점          → challenge

Priority 3 — 빌드 (가장 넓은 키워드):
  만들어/구현/개발/빌드/작성         → forge (복잡도 분기 적용)
  전체 파이프라인/스펙부터            → smelt (미구현 시 mold 제안)
```

## 복잡도 기반 forge vs mold 분기

Priority 3에서 "만들어/빌드" 키워드가 감지되면 대상 복잡도를 추가 판단한다:

| 입력 예시 | 판단 기준 | 라우팅 |
|-----------|-----------|--------|
| "결제 API 만들어줘" | 단일 파일/함수 수준 | forge 직행 (HIGH) |
| "결제 시스템 만들어줘" | 모듈/서비스 수준 | 2개 후보 제시 (MEDIUM) |
| "결제 플랫폼 전체 만들어줘" | 대규모/풀스택 | mold 또는 smelt 제안 |

기준:
- 단일 명사 (API, 함수, 컴포넌트, 훅) → forge
- 복합 명사 또는 "시스템/모듈/서비스" → mold 먼저 추천
- "전체/플랫폼/풀스택" → smelt (v2.1.0+) 또는 mold

## 신뢰도 판정 및 확인

```
HIGH   — 단일 의도 + 구체적 대상 → 즉시 라우팅 (확인 없이)
MEDIUM — 복수 의도 또는 모호     → 상위 2개 후보 제시, 사용자 확인
LOW    — 키워드 없음              → 전체 메뉴 표시
```

**HIGH 신뢰도**: 바로 실행하며 한 줄 이유를 출력한다:
```
security 실행 — SQL 인젝션 키워드 감지
```

**MEDIUM 신뢰도**: `AskUserQuestion`으로 1회 확인한다:
```
어떤 커맨드로 실행할까요?

> mold — 설계 먼저 (Recommended)
> forge — 바로 구현
```

**LOW 신뢰도**: 전체 메뉴를 표시한다:
```
어떤 작업인지 좀 더 구체적으로 알려주시거나 커맨드를 직접 선택하세요.

> cast    — 기획·방향 정제
> mold    — 기술 구조 설계
> forge   — 코드 구현
> assay   — 코드 품질 리뷰
> security — 보안 취약점 점검
> challenge — 적대적 리뷰
> debug   — 에러 분석
```

## 라우팅 학습

사용자가 라우팅 결과를 거부하면 보정 사례를 저장한다:

```bash
# .crb/routing-corrections.jsonl에 한 줄 append
{"query":"<원본 입력>","wrong":"<잘못 라우팅한 커맨드>","correct":"<사용자가 선택한 커맨드>","timestamp":"<ISO8601>"}
```

다음 라우팅 시 저장된 사례 상위 20개를 라우팅 판단 프롬프트의 컨텍스트로 포함한다:
```
과거 보정 사례를 참고해서 커맨드를 결정해:
  - "인증 관련 설계" → security로 잘못 라우팅됨, 정답은 mold
  - "로그인 플로우 구조" → security로 잘못 라우팅됨, 정답은 mold
  ...
  입력: "<현재 쿼리>"
```

상위 20개만 사용 (오래된 항목 자동 제외). LLM이 의미 기반으로 보정을 적용한다.

## 라우팅 결과 실행

커맨드가 결정되면 해당 커맨드를 직접 실행한다. auto는 라우터 역할만 하며 별도 출력을 저장하지 않는다.

라우팅 결과와 이유는 `.crb/runs/run-log.jsonl`에 기록한다:
```json
{"timestamp":"<ISO8601>","command":"auto","routed_to":"security","confidence":"HIGH","reason":"SQL 인젝션 키워드","query":"<원본 입력>","correction":false}
```

## Gotchas

- HIGH 신뢰도에서도 한 줄 이유는 반드시 출력한다 — 사용자가 이유를 알 수 있어야 함
- 라우팅이 불확실하면 MEDIUM으로 내려 확인받을 것 — HIGH 남발 금지
- smelt는 v2.1.0 전까지 라우팅 후보에서 제외
- 보정 사례는 완전히 틀린 경우에만 저장 — 사용자가 확인을 취소한 경우는 저장하지 않음
- auto 자체의 출력 파일은 생성하지 않는다 — 실행된 커맨드가 결과물을 생성
